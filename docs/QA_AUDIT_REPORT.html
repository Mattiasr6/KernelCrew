1. Hallazgos Críticos

Problemas que causan crashes, data leaks, o features completamente rotas. Deben resolverse antes del próximo deploy.

🔴 Backend — Métodos undefined: hasRole() y assignRole()

B1 · Crítico Spatie removido pero hasRole()/assignRole() siguen usándose
backend/app/Http/Controllers/AuthController.php:72
backend/app/Http/Controllers/CourseController.php:80, 275, 341
backend/app/Http/Requests/StoreCourseRequest.php:20
Las tablas de Spatie laravel-permission fueron dropeadas en la migración 2026_04_23_000001_drop_spatie_permission_tables.php. El trait HasRoles ya no se importa en el modelo User. Sin embargo, 4 archivos llaman a $user->hasRole() y $user->assignRole(), lo que causa Error: Call to undefined method en runtime.
Fix: Reemplazar $user->hasRole('admin') por $user->isAdmin() (método existente en User.php) y $user->assignRole('student') por $user->update(['role_id' => 3]).

🔴 Backend — Cursos draft visibles públicamente

B2 · Crítico CourseController::show() no aplica scope published()
backend/app/Http/Controllers/CourseController.php:213
La ruta GET /api/v1/courses/{id} es pública y no verifica si el curso está publicado. Cualquier usuario no autenticado puede ver cursos con status = 'draft' navegando por ID.
Fix: Agregar ->published() condicionalmente para usuarios no-admin/instructor.

🔴 Backend — Reviews de cursos draft expuestos

B3 · Crítico CourseReviewController::index() no verifica estado del curso
backend/app/Http/Controllers/Api/V1/CourseReviewController.php:20
La ruta pública GET /api/v1/courses/{id}/reviews usa Course::findOrFail($courseId) sin filtrar por published. Las reviews de cursos no publicados son accesibles.
Fix: Agregar ->published() al query o verificar $course->isPublished().

🔴 Frontend — Ruta /profile no existe

F1 · Crítico El componente Profile existe pero NO está registrado en app.routes.ts
frontend/src/app/features/layout/components/navbar.component.ts:137
frontend/src/app/features/profile/profile.component.ts (308 líneas implementadas)
El navbar tiene routerLink="/profile" pero app.routes.ts no tiene ruta para /profile. Al hacer clic en "Mi Perfil", el router cae en el wildcard ** y redirige silenciosamente a /courses. El componente Profile está completamente implementado pero es inaccesible.
Fix: Agregar { path: 'profile', loadComponent: () => import('./features/profile/profile.component').then(m => m.ProfileComponent) } en app.routes.ts.

🔴 Frontend — Doble /v1 en 3 servicios (llamadas API fallan)

F2 · Crítico 3 servicios construyen URLs con doble /v1 → 404 en runtime
frontend/src/app/core/services/dashboard.service.ts:12
frontend/src/app/core/services/certificate.service.ts:12
frontend/src/app/core/services/application.service.ts:12
El environment ya define apiUrl: 'https://.../api/v1', pero estos servicios concatenan otra vez /v1: 
private baseUrl = `${environment.apiUrl}/v1/instructor/dashboard`;
Resultado: https://.../api/v1/v1/instructor/dashboard → 404.
Fix: Quitar el /v1 extra de la ruta, o migrar estos servicios a usar ApiService en lugar de HttpClient directo.

🔴 Frontend — 56 memory leaks (sin unsubscribe)

F3 · Crítico Cero patrones de cleanup en 56 .subscribe() calls
Ningún componente usa takeUntil(), takeUntilDestroyed(), DestroyRef, ni unsubscribe() en ngOnDestroy. Esto significa que al navegar entre vistas, las suscripciones quedan vivas, causando:
  • Memory leaks progresivos
  • Actualizaciones de señales/DOM en componentes destruidos
  • Llamadas API duplicadas al volver a una vista
Fix: Adoptar takeUntilDestroyed() (Angular 16+) en todos los componentes con .subscribe(). Los 3 mayores ofensores: course-player.component.ts (5), course-list.component.ts (4), instructor-courses.component.ts (3).

🔴 Frontend — Provider inexistente en Angular

F4 · Crítico provideBrowserGlobalErrorListeners no existe en Angular
frontend/src/app/app.config.ts:1
El import import { provideBrowserGlobalErrorListeners } from '@angular/core'; referencia una función que no existe en Angular. Esto fue probablemente alucinado por una IA. Causa error de compilación. Nota: La build pasó en sesiones anteriores, posiblemente porque es un warning tree-shaken, pero debe corregirse.
Fix: Eliminar el import y su uso. El proveedor correcto es provideZoneChangeDetection o ErrorHandler.

2. Hallazgos Alta Prioridad

Problemas que degradan la experiencia de usuario, causan datos incorrectos, o introducen riesgos de seguridad.

🟠 Backend

B4 · Alto /auth/me no devuelve role_id
backend/app/Http/Controllers/AuthController.php:180-191
El endpoint retorna role (string) pero no role_id (número). El frontend usa role_id para lógica condicional en guards y componentes.
Fix: Agregar 'role_id' => $user->role_id en la respuesta.
B5 · Alto Inconsistencia "básico" vs "basic" en control de acceso por nivel
backend/app/Http/Controllers/CourseEnrollmentController.php:43 vs SubscriptionAccess.php:84
El middleware SubscriptionAccess maneja ambos 'basic' y 'básico', pero CourseEnrollmentController solo maneja 'básico'. Si un plan tiene nombre en inglés, el middleware deja pasar pero el enrollment controller lo rechaza.
Fix: Unificar usando in_array($planName, ['basic', 'básico']) en ambos lugares.
B6 · Alto toggleStatus() sin validación — bug de type-casting
backend/app/Http/Controllers/Admin/AdminUserController.php:215-219
(bool) "false" evalúa a true en PHP (string no vacío). Enviar {"is_active": "false"} activa al usuario en vez de desactivarlo. Sin validación.
Fix: Agregar $request->validate(['is_active' => 'required|boolean']);.
B7 · Alto Estadísticas de pagos ignoran filtros aplicados
backend/app/Http/Controllers/Admin/AdminPaymentController.php:49-51
El controller construye un query filtrado pero calcula total y count desde todos los pagos completados, ignorando fecha/método/búsqueda. Admin ve stats inconsistentes.
Fix: Clonar el query filtrado para las estadísticas.
B8 · Alto Respuestas de error sin campo success
backend/app/Http/Controllers/InstructorApplicationController.php:40, 44, 94
Los errores retornan ['message' => '...'] sin 'success' => false. Rompe el contrato de API que el frontend espera.
Fix: Agregar 'success' => false a todas las respuestas de error.
B9 · Alto N+1 queries en Dashboard del Instructor (60+ queries extra)
backend/app/Http/Controllers/Instructor/InstructorDashboardController.php:52-58, 76-83
Itera sobre cursos con $course->reviews()->avg('rating') y $course->enrollments()->count() dentro de loops. Con 20 cursos = ~60 queries extra.
Fix: Usar withAvg('reviews', 'rating') y withCount(['enrollments', 'enrollments as completed_count' => fn]) en el eager load.

🟠 Frontend

F5 · Alto 56 .subscribe() sin catchError — errores silenciados
56 instancias en 20+ archivos
El único catchError en todo el código está en auth.interceptor.ts para 401. Errores 5xx, network failures, y CORS son silenciados. Componentes con error: () => {} vacío no dan feedback visual al usuario.
Fix: Implementar un interceptor global de errores HTTP y usar NotificationService para errores 500+.
F6 · Alto Subscription component sin estado de carga inicial
frontend/src/app/features/student/student-subscriptions.component.ts:202
isLoading signal declarado pero nunca usado en el template. En conexiones lentas, el usuario ve una página vacía antes de que carguen los planes.
Fix: Agregar @if (isLoading()) { ... spinner/skeleton ... } en el template.
F7 · Alto Guards duplicados con lógica de redirect diferente
frontend/src/app/core/guards/auth.guard.ts:32-53 vs admin.guard.ts / instructor.guard.ts
adminGuard existe en dos archivos: uno redirecta a /dashboard, otro a /. Los imports en app.routes.ts usan los correctos, pero los duplicados generan confusión.
Fix: Eliminar los guards duplicados en auth.guard.ts (líneas 32-53) — son dead code.
F8 · Alto 6 enfoques CSS distintos en el mismo codebase
  • Tailwind utility classes (la mayoría)
  • Hex colors hardcodeados en styles: [...] arrays (30+ componentes)
  • CSS custom properties (var(--bg-primary)) — solo dashboard.component.ts
  • SCSS externo (admin-layout.component.scss)
  • Inline style="..." attributes
  • @apply dentro de styles: [...] (instructor-layout)
Cambiar el color de fondo requiere find-and-replace en 30+ archivos.
Fix: Extraer colores a variables CSS globales o extender el tema de Tailwind. El DESIGN.md ya define la paleta oficial.

6. Recomendaciones de Arquitectura

ARQ-1Centralizar tema CSS según DESIGN.md
Problema: 6 enfoques CSS distintos. Colores definidos en 30+ archivos con hex hardcodeados.
Solución: Extender tailwind.config.js con la paleta de DESIGN.md (zinc-950, cyan-500, emerald-500, rose-500, amber-500). Usar solo clases Tailwind en templates. Mover estilos de componentes a @apply en styles globales.
ARQ-2Estandarizar formato de respuesta API
Problema: Algunos endpoints retornan {success, data, message}, otros solo {message}. El campo data a veces es {user: ...}, a veces el objeto directo.
Solución: Crear un ApiResponse trait/macro en Laravel que garantice el formato {success: bool, message: string|null, data: mixed, errors: mixed|null} en TODOS los endpoints.
ARQ-3Adoptar takeUntilDestroyed() en todos los componentes
Problema: 56 memory leaks por falta de cleanup de subscriptions.
Solución: Angular 16+ ofrece takeUntilDestroyed(). Agregar a todos los .subscribe(): 
.pipe(takeUntilDestroyed()).subscribe(...)
O usar DestroyRef + takeUntil para versiones anteriores.
ARQ-4Migrar servicios a ApiService (eliminar HttpClient directo)
Problema: 3 servicios usan HttpClient directo con URLs manuales, causando el bug de doble /v1.
Solución: Todos los servicios deben inyectar ApiService que ya maneja environment.apiUrl como prefijo.
ARQ-5Extraer componentes compartidos (Sidebar, StatusBadge)
Problema: Sidebar duplicado 3 veces. Badge de estado duplicado en múltiples componentes.
Solución: Crear StudentSidebarComponent y StatusBadgeComponent en shared/.
ARQ-6Implementar ErrorInterceptor global
Problema: Solo se maneja 401. Errores 500, 403, 422 no tienen manejo centralizado.
Solución: Crear error.interceptor.ts que capture todos los errores HTTP y:
  • 500 → Toast "Error del servidor"
  • 403 → Toast "No tienes permisos"
  • 422 → Dejar que el componente maneje validation errors
  • 0 (network) → Toast "Sin conexión"
